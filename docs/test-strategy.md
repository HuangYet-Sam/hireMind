## 1. 文档目的

定义 HireMind MVP 阶段的测试策略、质量标准和验收流程，确保"候选人管理 + AI 画像 + 职位管理"最小闭环的质量可度量、可追溯。

## 2. 测试范围

### 2.1 MVP 模块覆盖

| 模块 | 子功能 | 测试优先级 |
|------|--------|-----------|
| 候选人管理 | CRUD、简历上传、标签管理 | P0 |
| AI 画像 | 论文式画像、AI 隐性推理、引用溯源 | P0 |
| 职位管理 | 职位 CRUD、候选人-职位匹配（子功能） | P0 |

### 2.2 非功能测试范围

| 维度 | 覆盖内容 | 优先级 |
|------|---------|--------|

## 3. 测试层级

### 3.1 单元测试

- **工具**：Vitest + @vue/test-utils（Vue 3 + FastAPI 生态）
- **覆盖率目标**：核心业务逻辑 >= 80%
- **重点覆盖**：
  - 简历解析逻辑
  - AI 画像数据组装逻辑
  - 候选人-职位匹配算法
  - 标签管理 CRUD 逻辑

### 3.2 集成测试

- **工具**：Vitest + SQLAlchemy Test Utils（SQLite内存数据库）
- **覆盖范围**：
  - API 路由与数据库的交互
  - AI 服务调用链路（OpenAI API 调用、Embedding 生成）
  - 文件上传与存储链路
- **关键约束**：AI 调用层必须 mockable，测试不依赖真实 API 调用

### 3.3 E2E 测试

- **工具**：Playwright
- **覆盖范围**：
  - 核心用户流程（创建职位 -> 导入候选人 -> AI 画像 -> 查看匹配）
  - 关键页面交互（画像详情页、匹配结果页）
- **执行时机**：CI/CD 流水线、发布前回归
- **MVP 阶段策略**：E2E 以手工验证为主，Playwright 框架搭建放到自动化路线图 Phase 2，降低 MVP 交付风险

### 3.4 AI 功能专项测试

> 这是 HireMind 的核心差异化，需要独立的测试维度和量化基线。

详见第 4 节。

## 4. AI 推理质量测试

### 4.1 事实性数据提取准确性

**定义**：AI 从简历中提取的结构化数据是否与原文一致。

**测试方法**：
- 准备标准测试简历集（至少 20 份，覆盖不同格式、语言、行业）
- 对每份简历验证：姓名、工作经历、教育背景、技能标签等字段
- 计算字段级准确率 = 正确提取字段数 / 应提取字段总数

**量化基线**：

| 指标 | 目标值 | 可接受值（MVP 首版） |
|------|--------|----------------------|
| 核心字段准确率（姓名、工作经历、教育背景） | >= 95% | >= 85% |
| 扩展字段准确率（技能标签、项目经历） | >= 90% | >= 80% |

容错标准：日期格式差异不算错误，但语义偏差算错误

### 4.2 AI 隐性推理可信度

**定义**：AI 基于简历事实做出的推理判断是否有依据、是否为幻觉。

**测试方法**：
- 构建推理测试用例集（至少 30 个推理场景）
- 每个推理结果必须满足：
  - 有明确的前提事实支撑
  - 推理链路可追溯（前提 -> 推理 -> 结论）
  - 结论不超出前提的合理推断范围
- 人工评审推理质量，分类为：合理推理 / 过度推理 / 幻觉

**量化基线**：

| 指标 | 目标值 | 可接受值（MVP 首版） |
|------|--------|----------------------|
| 合理推理比例 | >= 95% | >= 90% |
| 幻觉比例 | <= 5% | <= 10% |
| 过度推理（有部分依据但过度延伸） | <= 5% | <= 10% |

注：CEO 和 CTO 共同确认，MVP 首版采用可接受值，上线后根据实际数据迭代收紧到目标值。

**幻觉检测关键规则**：
- AI 不得编造简历中未提及的工作经历
- AI 不得推断未提及的技能（除非有明确的关联证据，如"用了 React"可推断"了解 JavaScript"）
- AI 不得对候选人的性格、价值观做无依据的判断

### 4.3 引用溯源完整性

**定义**：AI 画像中的每条信息是否都能追溯到简历原文的具体位置。

**测试方法**：
- 对每条画像信息验证：
  - 是否有引用标记（类似论文引用 [1][2]）
  - 点击引用标记是否能跳转到简历原文对应位置
  - 引用的原文内容是否与画像描述语义一致
- 引用覆盖率 = 有引用的画像信息数 / 画像信息总数

**量化基线**：
- 事实性信息引用覆盖率 = 100%（每条事实必须有出处）
- 推理性信息引用覆盖率 >= 80%（推理前提必须有出处，推理本身可无直接出处但需标注"AI推理"）
- 引用跳转可用率 = 100%（有引用标记的必须能跳转）

### 4.4 AI 推理标注规范

**视觉标记验证**（配合 Design System）：
- AI 推理结果必须标注"AI推理"标签，与事实数据视觉区分
- 标注样式需通过视觉回归测试，确保不遗漏
- 配色方案遵循 Design System 的 AI 标记色规范（紫色系）

## 5. 架构级测试：零侵入验证

### 5.1 AI 调用层可插拔性

**验证目标**：AI 服务层可独立禁用，不影响核心业务流程。

**测试用例**：
- 禁用 AI 服务后，候选人 CRUD 是否正常工作
- 禁用 AI 服务后，职位管理 CRUD 是否正常工作
- AI 服务超时/降级时，页面是否有合理的 fallback 展示
- 切换 AI 供应商（如从 OpenAI 切换到其他）是否只需修改配置

### 5.2 耦合度检测

- 代码审查时检查：业务逻辑代码中不应直接调用 AI API
- AI 调用必须通过抽象层（Service Interface）
- 集成测试中 AI 服务必须可 mock

## 6. Bug 管理规范

### 6.1 Bug 生命周期

```
发现 -> 确认 -> 分配 -> 修复 -> 验证 -> 关闭
                \                    ^
                 -> 延期 / 拒绝
```

### 6.2 严重程度定义

| 等级 | 定义 | 示例 | 响应时间 |
|------|------|------|---------|
| P0 阻塞 | 核心流程无法使用 | 候选人创建失败、AI 画像完全不可用 | 立即修复 |
| P1 严重 | 核心功能异常但有 workaround | AI 画像引用跳转失效、简历解析大面积错误 | 24h 内 |
| P2 一般 | 非核心功能异常 | 标签筛选不精确、分页异常 | 当前迭代内 |
| P3 轻微 | 体验问题 | 文案错误、样式微调 | 下个迭代 |

### 6.3 Bug 报告模板

```
**问题描述**：[具体现象描述]

**复现步骤**：
1. [步骤一]
2. [步骤二]
3. [步骤三]

**期望结果**：[应该怎样]

**实际结果**：[现在怎样]

**严重程度**：P[0-3]

**环境信息**：浏览器 / 设备 / 账号角色

**截图/录屏**：[如有]
```

## 7. 质量门禁

### 7.1 代码合并门禁

- 单元测试全部通过
- 新代码覆盖率不低于模块平均水平
- ESLint / TypeScript 检查无错误
- CTO 审查 + 自测清单确认（一人公司场景下不设多人 Code Review）

### 7.2 发布门禁

- E2E 核心流程全部通过
- AI 推理质量测试达到量化基线
- 无 P0 / P1 未关闭 Bug
- P2 Bug 数量不超过 5 个

### 7.3 AI 功能额外门禁

- AI 推理幻觉比例达标
- 引用覆盖率达标
- 零侵入验证通过（AI 层可独立禁用）
- AI 推理标注完整性检查通过

## 9. 数据安全测试

### 9.1 测试背景

简历属于敏感个人信息（包含姓名、联系方式、工作经历等），HireMind 作为 HR 系统必须确保数据安全。

### 9.2 测试范围

| 测试项 | 验证内容 | 优先级 |
|--------|---------|--------|
| 文件存储访问控制 | 简历文件是否通过签名 URL 访问，不可直接猜测路径下载 | P0 |
| API 接口鉴权 | 所有 API 端点是否要求有效 Token，未授权请求返回 401 | P0 |
| 数据隔离 | 不同租户/用户的数据是否严格隔离，不可越权访问 | P0 |
| 测试数据脱敏 | 测试环境不得使用真实个人信息，必须使用脱敏或合成数据 | P1 |
| 文件上传安全 | 限制文件类型（仅 PDF/DOCX）、文件大小、文件内容检测 | P1 |
| 敏感数据传输 | API 是否强制 HTTPS，简历上传是否加密传输 | P1 |

### 9.3 安全测试用例示例

- 未登录状态访问候选人列表 API，期望返回 401
- 直接通过 URL 拼接尝试下载他人简历文件，期望返回 403
- 上传 .exe 文件伪装为 .pdf，期望被拒绝
- 测试环境中检查是否有真实姓名、手机号等敏感数据残留

## 11. AI 推理类型分层测试

> 背景：TAD SQLAlchemy Models 中 AiInference 模型采用双字段设计（CEO 最终决策）：
> - type: InferenceNature 枚举（FACT / REASONING）— 决定视觉模式（事实/推理）
> - category: InferenceCategory 枚举（SKILL_TAG / EXPERIENCE_SUMMARY / CULTURE_FIT / MATCH_SCORE / SALARY_PREDICTION）— 决定业务语义颜色
>
> 两个维度独立：同一业务语义（如 SKILL_TAG）可以同时存在事实提取和隐性推理。测试按 type 和 category 两个维度独立覆盖。

### 11.1 事实提取（type = FACT）测试

- **定义**：从简历原文中直接提取的结构化数据
- **验收标准**：准确率 >= 95%（目标值），>= 85%（MVP 可接受值）
- **引用要求**：100% 必须有出处，无引用视为缺陷
- **测试重点**：字段级精度，不允许语义偏差
- **错误类型**：错提（张冠李戴）、漏提（存在但未提取）、乱提（无中生有）
- **适用 category**：所有 5 个业务语义均可产生事实提取
- **视觉标记**：灰蓝色系/中性色（事实提取不使用 AI 专属紫色，传达"客观事实"）

### 11.2 隐性推理（type = REASONING）测试

- **定义**：基于简历事实做出的推断、分析和评估
- **验收标准**：幻觉比例 <= 5%（目标值），<= 10%（MVP 可接受值）
- **引用要求**：推理前提必须有出处，推理本身标注"AI推理"
- **测试重点**：推理链路合理性，幻觉检测，reasoningChain 必填
- **错误类型**：过度推理、无依据推理、事实性幻觉
- **适用 category**：所有 5 个业务语义均可产生隐性推理
- **视觉标记**：紫色系（AI 专属色，传达"AI 的判断"）+ Sparkles 图标

### 11.3 类型标注验证（type 字段）

- 每条 AI 推理结果必须正确标注 type（FACT 或 REASONING）
- FACT 类型不得包含推理内容
- REASONING 类型必须有明确的前提事实引用和 reasoningChain
- type 字段标注错误视为 P1 Bug（用户会将 AI 推理误认为事实数据）

### 11.4 业务语义验证（category 字段）

- 每条 AI 推理结果必须正确标注 category
- 5 个 category 值的正确性验证：SKILL_TAG / EXPERIENCE_SUMMARY / CULTURE_FIT / MATCH_SCORE / SALARY_PREDICTION
- category 决定视觉呈现的颜色系，错误标注导致视觉混淆

### 11.5 组合覆盖矩阵

| type \ category | SKILL_TAG | EXPERIENCE_SUMMARY | CULTURE_FIT | MATCH_SCORE | SALARY_PREDICTION |
|----------------|-----------|--------------------|----|----|----|
| FACT | 事实提取：技能标签 | 事实提取：经历摘要 | 事实提取：文化背景 | 事实提取：匹配数据 | 事实提取：薪资数据 |
| REASONING | 隐性推理：技能推断 | 隐性推理：能力推断 | 隐性推理：文化匹配 | 隐性推理：匹配分析 | 隐性推理：薪资预测 |

测试时每个格子的验收标准不同：FACT 行按 11.1 标准，REASONING 行按 11.2 标准。

### 11.6 降级场景验收条件

1. AI 服务完全不可用 -- 页面正常渲染，AI 区块显示虚线占位 + "AI 分析暂不可用"
2. AI 服务超时（>10s）-- 先展示骨架屏 + "AI 分析生成中"，超时后降级为占位提示
3. AI 服务异步重试成功 -- 占位提示恢复为正常 AI 内容
4. AI 数据部分返回 -- 已有数据正常展示，缺失数据部分显示占位提示

### 11.7 分层统计

按 type 分组统计质量指标：
- FACT 组：准确率、引用覆盖率
- REASONING 组：幻觉比例、过度推理比例、推理前提引用覆盖率

按 category 分组统计可辅助分析 AI 在不同业务语义上的表现差异。
