## 7.3 实体关系图

```
┌──────────────┐
│   部门        │  ← 第八轮研讨新增（D1实体，P0）
│(departments) │     自引用树形结构(parent_id)
└──────┬───────┘     岗位必须归属部门
       │
       │ 1:N
       ▼
┌─────────────┐          ┌──────────────┐          ┌─────────────┐
│   岗位画像   │◄───1:N──│   匹配记录    │──N:1───►│  人才画像    │
│ (positions) │          │  (matches)   │          │(candidates) │
└──────┬──────┘          └──────────────┘          └──────┬──────┘
       │                                                  │
       │ 1:N                                              │ 1:N
       ▼                                                  ▼
┌─────────────┐                                   ┌──────────────┐
│   面试记录   │                                   │    简历      │
│(interviews) │                                   │  (resumes)   │
└─────────────┘                                   └──────┬──────┘
                                                        │
                                                        │ 1:N
                                                        ▼
                                                 ┌──────────────┐
                                                 │  画像更新     │
                                                 │(profile_     │
                                                 │  updates)    │
                                                 └──────────────┘
```

**关系说明**：

| 关系 | 类型 | 说明 |
|------|------|------|
| departments → positions | 1:N | 岗位必须归属部门（department_id FK必填） |
| positions ← matches → candidates | N:1 ← → N:1 | 匹配记录是连接岗位和候选人的对称关系，UNIQUE(position_id, candidate_id) |
| positions → interviews | 1:N | 面试必须关联岗位（position_id FK） |
| candidates → interviews | 1:N | 面试必须关联候选人（candidate_id FK） |
| candidates → resumes | 1:N | 一个候选人可以有多份简历 |
| candidates → profile_updates | 1:N | 画像更新溯源记录 |

## 7.4 招聘全链条8阶段流转

| 阶段 | 岗位 | 人才画像 | 简历 | 面试 | 匹配 |
|------|------|---------|------|------|------|
| A:需求 | ●模糊→精确定义 | ○无 | ○无 | ○无 | ○无 |
| B:对齐 | ●结构化→发布 | ○隐式理想型 | ○无 | ○计划制定 | ○无 |
| C:发布 | ●向量化→匹配池 | ○个别种子 | ○涌入 | ○无 | ○无 |
| D:投递 | ○不变 | ●构建/增量更新 | ●关联岗位→验证 | ○可能电面 | ●创建→冷启动匹配 |
| E:初筛 | ○可能调整 | ●轮廓形成 | ●交叉验证 | ○无 | ●评分筛选→推荐 |
| F:面试 | ○可能修正 | ●膨胀（数据黄金矿） | ●交叉验证 | ●数据生成 | ●状态推进 |
| G:Offer | ●锁定 | ●核心决策（不看简历） | ○退居 | ●历史依据 | ●结果确定 |
| H:入职 | ●在岗 | ●持续演进 | ○归档 | ●校准模型 | ●onboarded+`onboarded_at`记录+其他岗位withdrawn |

**关键节点**：
- **阶段D**是数据汇聚起点：简历→画像→匹配三条线同时激活
- **阶段F**是数据最丰富阶段：面试同时产生画像更新和岗位修正信号
- **阶段G**完全基于画像决策，简历退居

## 7.5 反向流转4种模式

### 模式1：人才驱动岗位（反向匹配）

- **触发**：HR搜索"这个候选人适合什么岗位"
- **数据流**：人才画像向量 → pgvector检索(position_embeddings) → 匹配岗位列表
- **标识**：`matches.initiated_by = 'candidate'`

### 模式2：面试数据回流修正岗位

- **触发**：连续N位候选人在同一维度失败（Cron分析任务检测）
- **数据流**：面试通过率统计 → NLP失败原因分析 → 岗位要求合理性建议 → HR确认调整
- **后续**：岗位向量重新生成 → 重新匹配

### 模式3：跨岗位候选人流转

- **触发**：面试发现候选人更适合其他岗位
- **数据流**：面试反馈 → 自动搜索画像vs所有开放岗位 → 推荐更匹配岗位

### 模式4：人才库激活（被动→主动）

- **触发**：Cron每周扫描沉睡候选人（90天无互动）
- **数据流**：被动候选人画像向量 vs 所有open岗位 → 发现新匹配 → 推荐HR激活

## 7.6 四层反馈闭环

| 闭环层 | 触发频率 | 涉及实体 | 数据流向 | 实现阶段 |
|--------|---------|---------|---------|---------|
| L1 数据反馈 | 实时 | 简历→画像→匹配 | 单向管道 | Phase 1 (M1-M3) |
| L2 流程反馈 | 天级 | 面试→画像+岗位 | 双向更新 | V1.0 (M4-M6) |
| L3 运营反馈 | 周级 | 匹配精度+通过率 | 统计反馈 | V1.5 (M7-M12) |
| L4 战略反馈 | 季度 | 全链路数据 | 战略调整 | V2.0 (M13-M18) |

**闭环详细说明**：

### L1 数据反馈（实时，Phase 1）

简历上传 → 解析 → 画像构建 → 自动匹配 → 匹配分数更新。这是单向管道，数据从简历流向匹配结果。

### L2 流程反馈（天级，V1.0）

面试反馈双向更新：
- **方向1**：面试评分/评价 → 更新候选人画像（技能确认/新技能发现/经验验证）
- **方向2**：连续面试失败信号 → 岗位要求修正（如要求过高/技能不合理）

### L3 运营反馈（周级，V1.5）

匹配精度统计 → 通过率分析 → 权重/阈值调整。例如：某岗位匹配Top10候选人面试通过率仅20% → 调整匹配维度权重。

### L4 战略反馈（季度，V2.0）

全链路数据分析 → 招聘策略调整。例如：某类岗位长期招不到人 → 分析市场供需/薪酬竞争力/雇主品牌。
