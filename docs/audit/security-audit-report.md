### 🔴 CRITICAL — 致命风险（必须立即修复）

#### C-01: 候选人PII明文存储，零加密保护

**风险等级**：🔴 CRITICAL
**PRD要求**：§11.2 数据分类分级 — L3敏感数据（候选人姓名、手机、邮箱、身份证号）需AES-256加密存储
**当前状态**：

`prisma/schema.prisma` 中候选人模型：
```prisma
model Candidate {
  name         String       // 明文存储 ❌
  email        String?      // 明文存储 ❌
  phone        String?      // 明文存储 ❌
  // 无 email_hash 字段 ❌
  // 无 phone_hash 字段 ❌
}
```

**合规违规**：
- 违反《个人信息保护法》第51条：未采取相应加密措施保护敏感个人信息
- 违反《个人信息安全规范》(GB/T 35273) 6.3条：个人敏感信息需加密存储
- 数据库泄露时，候选人姓名+手机+邮箱全部暴露

**修复建议**：
1. `candidates.name` → `name_encrypted VARCHAR(512)`（AES-256密文）
2. `candidates.email` → `email_encrypted VARCHAR(512)` + `email_hash VARCHAR(64)`（SHA-256哈希，用于去重查询）
3. `candidates.phone` → `phone_encrypted VARCHAR(512)` + `phone_hash VARCHAR(64)`
4. 加密密钥通过 HashiCorp Vault 管理，按 `tenant_id` 隔离
5. 解密操作必须记录审计日志（`audit_logs`，source=`pii_decrypt`）

#### C-02: 认证体系完全缺失

**风险等级**：🔴 CRITICAL
**PRD要求**：§11.2 JWT Token 24h过期 + Refresh Token 7天 + RBAC
**当前状态**：

- `src/app/(auth)/login/page.tsx` — 仅 `<div>Login Page</div>` 空壳
- `src/app/(auth)/register/page.tsx` — 仅 `<div>Register Page</div>` 空壳
- 无认证中间件（搜索 `middleware` 结果为零）
- 无JWT库依赖
- 无RBAC权限矩阵
- 所有API端点（`/api/jobs`、`/api/candidates`、`/api/ai/match`、`/api/ai/profile`、`/api/upload`）均无任何认证校验

```typescript
// src/app/api/candidates/route.ts — 无任何认证
export async function GET() {
  return NextResponse.json({ success: true, data: [] });
}
```

**影响**：
- 任何人可无条件访问所有API端点
- 无用户身份标识，多租户隔离无法实现
- 所有CRUD操作无法追踪操作者

**修复建议**：
1. 实现 FastAPI JWT 中间件（目标架构使用 Python + FastAPI）
2. 实现 RBAC 权限模型：Owner → Admin → Member → HRBP（预留）
3. 所有API端点统一通过 `_authorize()` 装饰器校验
4. JWT密钥存储于 HashiCorp Vault，90天轮换

#### C-04: 候选人数据生命周期管理缺失（被遗忘权）

**风险等级**：🔴 CRITICAL
**PRD要求**：§5.7 + §12.1 候选人可要求删除全部个人数据，系统7个工作日内完成
**当前状态**：

2. 简历上传/渠道导入前强制获取知情同意
3. 实现软删除 → 保留期（7天完成进行中流程）→ 硬删除（含pgvector向量+MinIO文件）三级级联
4. 已入职候选人仅可删除简历原文，保留脱敏画像（法律依据：劳动合同法第50条）

### 🟠 HIGH — 高风险

#### H-01: LLM数据脱敏Pipeline未实现

**风险等级**：🟠 HIGH
**PRD要求**：§5.7 P0-06 简历全文发送LLM API前PII自动脱敏
**当前状态**：`src/lib/ai/providers/openai.ts` 为空类，无脱敏逻辑

**影响**：
- 简历原文（含姓名、手机、邮箱、身份证号）可能以明文形式发送至 OpenAI API
- 违反数据最小化原则（个保法第6条）

1. 实现 §5.7 定义的脱敏Pipeline：姓名→`[NAME_N]`、手机→`[PHONE_N]`、邮箱→`[EMAIL_N]`、身份证→`[IDCARD_N]`
2. 反替换映射仅在内存中维护，请求完成后立即销毁
3. 每次LLM调用记录脱敏日志

**当前状态**：零实现

#### H-03: AI推理结果标注未实现

**风险等级**：🟠 HIGH
**PRD要求**：§5.7 AI推理区分事实数据vs AI推理（FACT vs REASONING）
**当前状态**：
- `prisma/schema.prisma` 定义了 `AiInference` 模型（含 `InferenceNature` 枚举：FACT/REASONING）
- 但Service层和AI层均为空壳，无实际标注逻辑

**影响**：
- 用户无法区分"简历中明确写的"和"AI推断的"
- AI幻觉（hallucination）可能被用户误认为事实，导致错误招聘决策
- 违反个保法第24条"自动化决策"透明度要求

**修复建议**：
1. 实现 FACT/REASONING 双路径标注
2. FACT：实色背景 + 引用标记，准确率≥95%
3. REASONING：渐变背景 + 置信度标签，幻觉率≤10%，必须含 `reasoningChain`
4. 前端组件 `inferenceBadge.tsx`（已有空壳）实现视觉区分

#### H-06: 数据库连接无SSL/TLS

**风险等级**：🟠 HIGH
**PRD要求**：§11.2 全站HTTPS/TLS 1.3 + 数据库连接加密
**当前状态**：

`.env` 配置：
```
DATABASE_URL="postgresql://johndoe:randompassword@localhost:5432/mydb?schema=public"
```

无 `sslmode=require` 参数，数据库连接明文传输。

**影响**：
- 数据库凭据（`johndoe:randompassword`）为默认占位值
- 数据库连接未加密，网络嗅探可截获候选人PII数据
- 生产环境部署时极易遗漏SSL配置

**修复建议**：
1. `DATABASE_URL` 添加 `?sslmode=require`
2. 使用 HashiCorp Vault 管理数据库凭据
3. `.env` 不提交至版本控制（检查 `.gitignore`）
4. Caddy 反向代理启用自动 HTTPS

## 6. 总结与建议

### 6.1 核心判断

HireMind PRD V3.6 的安全设计在同类SaaS产品中属于**顶级水平**——覆盖了数据分级（L1-L4）、加密存储（AES-256）、密钥管理（Vault）、AI安全（脱敏Pipeline + Prompt注入防护 + 推理标注）、审计日志、CSRF防护等全方位安全需求，且严格对齐中国个保法合规要求。

**但所有安全设计均为"文档态"，代码实现为零。**

### 6.2 最大风险

1. **技术栈冲突**：当前 Next.js/TypeScript/Prisma 代码无法承载 PRD 定义的安全架构（Python FastAPI + SQLAlchemy + Hermes Agent），必须清零重建
2. **合规时钟**：一旦有真实候选人数据入库，个保法合规要求立即生效，当前系统完全无法满足
3. **AI安全**：AI能力是产品核心差异化，但脱敏Pipeline和Prompt注入防护不实现即不可上线

### 6.3 最终建议

1. **清零重建时将安全作为第一优先级**，不是"功能做完再加安全"，而是"安全与功能同步建设"
2. **C-01~C-04 四个Critical问题必须在M1里程碑前解决**
3. **H-01/H-02 两个AI安全问题必须在M2（AI核心）里程碑前解决**
4. **上线前完成个保法合规自查**，确保 consent_records + 删除权 + 审计日志 + 数据出境脱敏全部到位
