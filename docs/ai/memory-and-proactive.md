## 1. AI记忆系统（§5.5）

> **新增于**：第十三轮研讨 | **裁决#30** | **优先级**：⭐⭐⭐⭐⭐

利用Hermes Memory构建三层记忆体系，使Agent从"无状态工具"升级为"有记忆的AI同事"。

### 1.1 三层记忆架构

| 层级 | 内容 | 写入时机 | 读取场景 |
|------|------|---------|---------|
| **L1 用户偏好** | HR的决策模式（偏好什么候选人、常用筛选条件、薪资偏好范围） | HR做录用/拒绝决策时Agent主动询问；每周自动分析筛选模式提炼 | 匹配排序调整权重；JD/Offer生成参考 |
| **L2 组织知识** | 团队特性、面试官风格、薪资结构惯例、招聘周期规律 | HR主动告知；Agent从历史数据提炼 | 面试准备参考团队风格；Offer参考薪资惯例 |
| **L3 决策历史** | 每次重要决策（录用/拒绝/调薪）的原因记录 | 决策发生时自动记录 | 类似候选人出现时提供历史参考；AI推理时溯源 |

### 1.2 关键约束

- 记忆条目必须标注来源（用户主动告知 / AI自动提炼 / 决策记录）
- AI自动提炼的记忆需在AI Context Bar中展示，HR可确认或纠正
- 记忆过期策略：L1偏好3个月无使用自动降权；L3决策历史永久保留

### 1.3 冷启动策略（Cold-Start Strategy）

> **P2-14说明**：本节定义冷启动的**AI记忆采集机制**。冷启动的**用户引导流程**定义在§10.4新用户引导设计。§10.4的三步引导流程（创建岗位→上传简历→查看匹配）同时触发本节的画像采集+偏好提炼——Step 1触发L2组织知识采集，Step 2触发画像构建，Step 3触发L1偏好积累。

新租户首次使用时，Agent主动发起"组织知识采集"对话，快速构建初始记忆：

1. **组织画像采集**：询问公司规模/行业/招聘偏好 → 写入L2组织知识
2. **历史偏好提炼**（可选）：导入历史简历样本 → 提炼L1用户偏好（技能偏好/学历偏好/经验偏好）
3. **决策偏好积累**：首次匹配时主动询问决策理由 → 快速积累L1偏好

冷启动期目标：**7天内完成初始记忆构建**，之后Agent个性化推荐准确率显著提升。采集进度可在工作台Dashboard「AI洞察」区域查看，Agent会提示剩余采集项。

## 2. 主动式AI引擎（§5.6）

> **新增于**：第十三轮研讨 | **裁决#32** | **优先级**：⭐⭐⭐⭐⭐

基于Hermes Cron + Webhook构建9类主动式AI场景，使AI从"被动响应"升级为"主动行动"。

### 2.1 九类主动式AI场景

| # | 场景 | 触发方式 | 推送渠道 | 优先级 |
|---|------|---------|---------|--------|
| 1 | 新简历自动筛选推送 | Webhook（简历上传完成） | 微信+Web通知 | P0 |
| 2 | 面试前准备包推送 | Cron（面试前2h） | 微信+邮件 | P0 |
| 3 | 岗位超期预警 | Cron（每日8:00扫描） | 微信+工作台待办 | P0 |
| 4 | Offer跟进提醒 | Cron（每日9:00扫描） | 微信+工作台待办 | P1 |
| 5 | 招聘周报 | Cron（每周五17:00） | 邮件+微信 | P1 |
| 6 | 高匹配候选人推荐 | 回调（匹配完成且>90%） | 微信+Web通知 | P1 |
| 7 | 漏斗异常预警 | Cron（每日22:00分析） | 工作台+邮件 | P2 |
| 8 | 沉睡候选人激活 | Cron（每周一9:00扫描） | 工作台推荐 | P2 |
| 9 | 入职前关怀提醒 | Cron（入职前1天+Day7/30/60） | 微信+工作台待办 | P1 |

### 2.2 技术实现

- **Cron场景**：Hermes Cron Job → Agent → Service层 → send_message
- **Webhook场景**：外部系统 → `POST /webhooks/:name` → Agent → Service层
- **回调场景**：匹配完成 → Service层回调 → Agent → 推送通知

### 2.3 频率限制与成本上限（P2-24修复）

**系统级成本预算**：
- 每租户每月AI调用成本上限：**¥200/租户/月**（Phase 1免费版基准，专业版¥500，企业版按需配置）
- 成本计算口径：LLM Token消耗按实时价格折算（gpt-4.1-mini约¥0.01/次，gpt-4.1约¥0.05/次），每日累计
- 预算余量预警：消耗达80%时降频（高优场景降为每日一次），达95%时仅保留P0场景（场景1/2/3），达100%暂停所有主动式AI
- 预算配额存储：Redis `budget:{tenant_id}:monthly`，每月1日重置

**候选人去重窗口**：
- 同一候选人在**30天去重窗口**内不重复触发主动式推送（如场景6"高匹配推荐"对同一候选人30天内仅推送一次）
- 去重键：Redis `proactive:dedup:{tenant_id}:{candidate_id}:{scenario}`，TTL=30天
- P0场景（场景1/2/3）不受去重窗口限制，确保业务时效性

**批量处理优化**：
- Cron扫描类场景（3/4/5/7/8）采用批量处理：单次Cron扫描结果按优先级排序，Top 10条推送，其余写入工作台待办
- 单次Cron执行最大AI调用次数限制：**20次/Cron Job**，超出排队至下一周期

### 2.4 确认节点定义（Human-in-the-Loop，P0-08修复）

> ⚠️ **原则3边界强化**：Agent执行到关键节点必须暂停等待HR确认，采用"分段确认+关键节点暂停"模式。

| 类别 | 操作 | 是否可自主执行 | 需确认节点 | 说明 |
|------|------|:---:|:---:|------|
| 🔓 自主操作 | 简历解析/字段提取 | ✅ | — | 纯数据处理，无业务风险 |
| 🔓 自主操作 | 匹配计算/评分排序 | ✅ | — | 算法计算，结果可审阅 |
| 🔓 自主操作 | 信息查询/统计汇总 | ✅ | — | 只读操作 |
| 🔓 自主操作 | 画像构建/隐性推理 | ✅ | — | AI推理结果标注清晰 |
| 🔓 自主操作 | 日报/周报生成 | ✅ | — | 自动推送，无决策影响 |
| 🔒 确认操作 | 发布岗位到外部渠道 | ❌ | ✅需HR确认 | 影响公司对外形象 |
| 🔒 确认操作 | 推荐候选人给用人经理 | ❌ | ✅需HR确认 | 影响决策质量 |
| 🔒 确认操作 | 发送面试邀请给候选人 | ❌ | ✅需HR确认 | 直接接触候选人 |
| 🔒 确认操作 | 发送Offer给候选人 | ❌ | ✅需HR确认 | **最高风险**，法律效力 |
| 🔒 确认操作 | 修改薪资方案 | ❌ | ✅需HR确认 | 涉及财务承诺 |
| 🔒 确认操作 | 撤回Offer | ❌ | ✅需HR确认 | 法律与雇主品牌风险 |

**确认流程**：
1. Agent执行到🔒确认操作时，自动暂停任务链
2. Agent通过IM/Web推送确认请求给HR，附带操作摘要+影响说明
3. HR回复"确认"→ Agent继续执行；HR回复"拒绝"/"修改"→ Agent调整或终止
4. 超时规则：确认请求24h未响应自动提醒，48h未响应任务自动取消
5. 确认记录写入`audit_logs`，确保可追溯

## 3. AI任务中心（§4.0b）

**路由**：`/hr/tasks` | **新增于**：第十三轮研讨 | **优先级**：P0

> HR不需要知道Agent在后台做什么，但需要看到"AI在帮我做什么、做完了什么"。AI任务中心是Agent执行状态的**业务化可视化入口**，隐藏技术细节（Agent ID、Session），只展示业务语义。

### 3.2 支撑端点

| 端点 | 用途 |
|------|------|
| `GET /api/v1/tasks/running` | 运行中Agent任务列表（实时进度） |
| `GET /api/v1/tasks/completed` | 已完成任务（最近100条） |
| `GET /api/v1/tasks/queued` | 队列中Cron任务 |
| `POST /api/v1/tasks/:id/cancel` | 取消任务 |
| `POST /api/v1/tasks/:id/retry` | 重试失败任务 |
| `GET /api/v1/tasks/:id` | 任务详情（子步骤日志） |

### 3.3 底层映射

- **运行中** = `delegate_task`子Agent
- **已完成** = Cron历史
- **队列** = Cron待执行
